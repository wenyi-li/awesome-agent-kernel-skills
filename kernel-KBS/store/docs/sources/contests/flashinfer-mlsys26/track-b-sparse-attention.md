# Track B: DeepSeek V3.2 Sparse Attention

## Problem Description

Two-stage sparse attention kernel for DeepSeek V3.2 models on B200 GPUs. The kernel must implement a lightning indexer to select relevant tokens, then perform Multi-head Latent Attention (MLA) over only the selected tokens.

**Benchmark identifiers**:
- Indexer: `fp8_h64_d128_topk2048_ps64`
- Attention: `h16_ckv512_kpe64_topk2048_ps64`

## Two-Stage Architecture

### Stage 1: Lightning Indexer

FP8 scorer that selects the top-2048 most relevant tokens per query from the full KV cache:

```
Input: query (h=64 heads, d=128 per head)
KV cache: all tokens in sequence
Output: indices of top-2048 tokens per query

// Block-level scoring:
// 1. Compute approximate attention scores using FP8 for speed
// 2. Score at block granularity (block_size=64)
// 3. Select top-2048/64 = 32 blocks per query
// 4. Return token indices for selected blocks
```

Parameters: h=64 heads, d=128 head dimension, topk=2048 tokens, page_size=64.

### Stage 2: Sparse MLA Attention

Full-precision MLA attention over only the 2048 selected tokens:

```
Input: query, selected KV tokens (2048 per query)
MLA cache per token: 656 bytes
  - 512 bytes FP8 compressed KV (c_kv)
  - 16 bytes FP32 scale factors (4 scales for 512/128 blocks)
  - 128 bytes BF16 RoPE embeddings (k_pe)
Output: attention output

// MLA attention:
// 1. Decompress KV: k, v = project(c_kv, scales)
// 2. Apply RoPE: k_rope = k + k_pe
// 3. Compute attention: softmax(Q @ K^T / sqrt(d)) @ V
// 4. Only over 2048 selected tokens (not full sequence)
```

Parameters: h=16 heads (after TP=8 split, padded to 64), ckv=512, kpe=64, topk=2048, page_size=64.

## Key Challenges

### 1. Continuous Batching with Variable Sequence Lengths

In production serving, each request in a batch has a different sequence length. The indexer must handle variable-length sequences efficiently:

```
// Batch of requests with different context lengths:
// Request 0: 1024 tokens
// Request 1: 8192 tokens
// Request 2: 32768 tokens
// Each needs its own top-2048 selection
```

### 2. Kernel Padding Overhead

With TP=8, DeepSeek V3.2 has 16 attention heads per GPU, but hardware may require padding to 64:

```
// Physical heads: 16 (after TP=8 split)
// Padded heads: 64 (for efficient tensor core tiling)
// 4x compute waste on padding unless handled carefully
```

### 3. Variable Sparsity Patterns

The indexer produces different sparsity patterns for each query head and each batch element. The attention kernel must handle arbitrary index sets without assuming contiguous memory access.

### 4. Two-Kernel Coordination

The indexer and attention stages must coordinate:
- Indexer output (indices) becomes attention input
- Memory layout of indices affects attention memory access patterns
- Pipeline overlap between indexer and attention for different batch elements

### 5. MLA Cache Layout

Each token's MLA cache (656 bytes) has mixed data types:

```
struct MLAToken {
    uint8_t c_kv[512];      // FP8 compressed KV
    float   scales[4];       // FP32 block scales (512/128 = 4 blocks)
    __nv_bfloat16 k_pe[64]; // BF16 RoPE embeddings
};  // Total: 512 + 16 + 128 = 656 bytes
```

This mixed layout complicates vectorized loading and TMA descriptor setup.

## Optimization Strategies

### Indexer Optimizations

```
// FP8 approximate scoring for token selection
// Key insight: scoring doesn't need full precision
// FP8 matmul: query @ compressed_kv^T is sufficient for ranking

// Block-level scoring reduces computation:
// Instead of scoring 100K individual tokens,
// score 100K/64 = 1562 blocks, then select top-32 blocks
```

### Attention Optimizations

```
// Sparse attention over selected indices:
// 1. Gather selected KV tokens (indirect indexing)
// 2. Decompress FP8 -> BF16 on the fly
// 3. Standard flash attention over 2048 tokens
// 4. No causal mask needed (all selected tokens are relevant)

// Memory access pattern:
// Indices are sorted by block for spatial locality
// Prefetch next block of KV tokens while computing current
```

### Fusing Indexer + Attention

```
// Aggressive fusion: single kernel for both stages
// 1. Compute block scores (FP8 matmul)
// 2. Top-k selection (in shared memory)
// 3. Gather selected KV blocks
// 4. Full-precision attention
// Eliminates intermediate score buffer in HBM
```
