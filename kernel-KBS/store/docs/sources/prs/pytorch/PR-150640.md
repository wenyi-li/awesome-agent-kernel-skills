## Summary
Fixes incorrect kernel launch bounds for avgpool2d backward pass on Blackwell (sm100) and sm120 GPUs by replacing compile-time `__CUDA_ARCH__` preprocessor checks with runtime device property queries.

## Problem
The `__CUDA_ARCH__` macro is not visible in host code, which caused incorrect launch bounds to be calculated when compiling for Blackwell architecture. This resulted in "too many resources requested for launch" errors when running avgpool2d backward kernels on sm100 and sm120 GPUs. The original code used compile-time preprocessor directives that only work in device code, not in the host-side kernel launch configuration.

## Solution / Techniques
Replaced the compile-time `#if defined(__CUDA_ARCH__) && __CUDA_ARCH__ >= 1000` check with a runtime query using `at::cuda::getCurrentDeviceProperties()`. The fix detects whether the current device is sm100+ at runtime and adjusts the thread count from 1024 down to 768 for these architectures, which have different register file constraints.

## Key Code
```cuda
// Before (broken - __CUDA_ARCH__ not visible in host code):
#if defined(__CUDA_ARCH__) && __CUDA_ARCH__ >= 1000
constexpr int double_threads = 768;
#else
constexpr int double_threads = 1024;
#endif

// After (fixed - runtime device property query):
cudaDeviceProp* properties = at::cuda::getCurrentDeviceProperties();
const bool gesm10x = properties->major >= 10;
int double_threads = 1024;
if (gesm10x) {
  double_threads = 768;
}
```

## Performance
This is a correctness fix rather than a performance optimization. Without the fix, avgpool2d backward kernels fail to launch entirely on Blackwell GPUs due to requesting too many registers per thread block. The reduced thread count of 768 (from 1024) accounts for the different register allocation constraints on sm100/sm120 architectures.
