## Summary

## Motivation

Fixes #24738

When serving multimodal models, requests with bad/slow video URLs (e.g. unreachable hosts like `10.255.255.x` or very large files) block the engine's event loop via synchronous `future.result()` calls in `load_mm_data` / `fast_load_mm_data` / `legacy_load_mm_data`. This prevents the server from responding to health check requests, causing orchestrators (K8s, etc.) to restart the service unnecessarily.

## Changes

Convert `load_mm_data`, `fast_load_mm_data`, and `legacy_load_mm_data` in `base_processor.py` from synchronous to async functions. Replace blocking `future.result()` with `await asyncio.wrap_future(future)`, which yields control back to the event loop while waiting for the ThreadPoolExecutor futures to complete.

All processor subclass call sites are updated to `await self.load_mm_data(...)` accordingly (they are already inside `async def process_mm_data_async`).

## Test plan

- Reproduce with the issue's reproduction script: 15 concurrent requests with bad video URLs + health check
- Before fix: `curl /health --max-time 10` times out
- After fix: health check responds immediately regardless of pending resource fetches
