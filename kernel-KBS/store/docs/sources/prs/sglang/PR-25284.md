## Summary

## Motivation

Gemma4 shipped without pipeline-parallel support: launching either the LM (`Gemma4ForCausalLM`) or the multimodal entry (`Gemma4ForConditionalGeneration`) with `--pp-size 2` immediately crashed at with `AssertionError: Pipeline Parallel is not compatible with this model`. 

The runner gates PP on whether the model's `forward()` declares a `pp_proxy_tensors` parameter; neither Gemma4 entry did.

This becomes a real constraint on dense single-node setups. The 26B BF16 checkpoint occupies ~48 GB of weights on every TP rank, leaving little KV-cache headroom on an 80 GB H100. PP-sharding the layer stack across two ranks halves
the per-rank weight footprint and roughly doubles the available KV budget.


## Modification

### LM path (`gemma4_causal.py`)

`Gemma4TextModel` and `Gemma4ForCausalLM` follow the canonical sglang PP pattern
(see `apertus.py`):

- `Gemma4TextModel.__init__` stores `self.pp_group`, materializes
  `embed_tokens` / per-layer-embedding tables / `per_layer_model_projection` on
  the first rank only and `norm` on the last rank only (`PPMissingLayer` on the
  other ranks).  `make_layers(..., pp_rank, pp_size)` returns the layer slice
