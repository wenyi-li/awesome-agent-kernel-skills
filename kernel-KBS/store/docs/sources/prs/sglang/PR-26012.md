## Summary

## Motivation

`HiSparseCoordinator` was built in `ModelRunner` *after* `init_attention_backend()`, so the three DSV4 attention backends (`DeepseekV4AttnBackend`, `DeepseekSparseAttnBackend`, `DeepseekV4HipRadixBackend`) could not capture it at `__init__` time. Each backend worked around this with a `@property` that reads `self.model_runner.hisparse_coordinator` on every call.

Now that pool refs are no longer threaded through `ForwardBatch` (landed in #25983), the same principle applies to `hisparse_coordinator`: it should be captured at construction time, not read lazily through `model_runner`.

## Modifications

- **`model_runner.py`**: Move the `if self.enable_hisparse:` initialization block to before `init_attention_backend()` in the cuda/musa init path. The only ordering constraint on `hisparse_coordinator` is that it precedes CUDA graph capture; there is no dependency on `kernel_warmup`.

- **`deepseek_v4_backend.py`, `dsa_backend.py`, `deepseek_v4_backend_hip_radix.py`**: Replace the 5-line `@property hisparse_coordinator` (lazy forwarder to `model_runner`) with a plain `self.hisparse_coordinator = model_runner.hisparse_coordinator` assignment in `__init__`.
