## Summary

## Motivation

`ForwardBatch` carried four runtime-ref fields — `req_to_token_pool`, `token_to_kv_pool`, `attn_backend`, `hisparse_coordinator` — that have nothing to do with the per-batch input data the dataclass exists for. They were threaded into every model-layer file (`forward_mha.py`, `forward_mla.py`, `qwen3.py`, `deepseek_v4.py`, …), every attention backend, every hand-rolled `ForwardBatch(...)` constructor, and every speculative-decoding path. Removing them lets `ForwardBatch` stay focused on actual batch state, makes the attention backend solely responsible for its own pool refs, and gives PDmux / TBO / spec a uniform way to swap the active backend.

## Modifications

- New `python/sglang/srt/model_executor/forward_context.py` exporting a frozen `ForwardContext` dataclass (single mandatory field `attn_backend`) plus a scoped `forward_context()` context manager. Convenience pool accessors `get_token_to_kv_pool()` / `get_req_to_token_pool()` derive from `get_attn_backend().*` — no separate pool global needed.
- Drop four fields from `ForwardBatch`: `req_to_token_pool`, `token_to_kv_pool`, `attn_backend`, `hisparse_coordinator`. Drop the matching assignments from `ForwardBat

## Problem

feat(model_runner): remove pool/backend refs from ForwardBatch via ForwardContext
