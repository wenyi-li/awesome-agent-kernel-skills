## Summary

Fix a bug where `rid_to_state` entries are not cleaned up when queued requests are aborted, causing `Duplicate request ID detected` errors on resubmission.

## Motivation
When aborting all requests (e.g., during weight updates or pause/resume), queued requests that were waiting in the scheduler are aborted and an `AbortReq` is sent back to the `TokenizerManager` via `_handle_abort_req`. However, this handler never called `del self.rid_to_state[rid]`, leaving stale entries behind.

The root cause is that **running** and **queued** requests follow completely different abort paths, and the queued path was missing cleanup:

  **Running request abort** — goes through the normal forward → output pipeline:
```text
  scheduler: req.to_finish = FINISH_ABORT()
    → next forward pass finishes normally
    → stream_output() sends BatchStrOutput
    → detokenizer forwards to tokenizer_manager
    → _handle_batch_output()
      → del self.rid_to_state[rid]   ✅ cleaned up
```

  **Queued request abort** — never enters forward, scheduler sends `AbortReq` directly:
```text
  scheduler: waiting_queue.pop(i)
    → sends AbortReq back to tokenizer_manager
    → _handle_abort_req(
