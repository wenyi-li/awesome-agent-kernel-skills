## Summary

## Motivation

#25985 fixed the Wan VAE correctness issue when Conv3d weights use `channels_last_3d`. This follow-up keeps that default path covered by CI and keeps the existing explicit environment override for experiments and emergency rollback.

The follow-up CI comparison showed that switching the default to a narrow qwen-only auto policy causes Wan/MOVA/ModelOpt consistency drift without a stable end-to-end speed win. Therefore the default remains enabled for CUDA/ROCm VAE/video_VAE Conv3d weights, matching the current ground truth revision. Non-CUDA/ROCm platforms keep the regular contiguous path because NPU does not support `contiguous(memory_format=torch.channels_last_3d)`.

## Modifications

- Route VAE loading and component accuracy reference loading through a shared `channels_last_3d` selector.
- Preserve the current CUDA/ROCm default behavior:
  - unset / `auto`: enable `channels_last_3d` for VAE/video_VAE Conv3d weights
  - `false`: explicitly disable it for targeted experiments or rollback
  - `true`: explicitly enable it
- Gate Wan VAE Conv3d input layout matching to CUDA/ROCm so NPU/CPU/MPS/XPU/MUSA do not call `channels_last_3d` contiguous.
