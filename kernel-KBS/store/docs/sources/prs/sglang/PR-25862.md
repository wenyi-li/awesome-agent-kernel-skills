## Summary

## Why req-pool-indexed

The ring allocator slot id is a temporary counter — slot K denotes different requests at different times, driving sizing miscalculations (chunked-prefill blowup), wrap-around aliasing, slot-reuse staleness, and a cross-stream indices-tensor lifetime race. Indexing every future-map buffer by `req_pool_idx` instead binds each slot to a request's lifetime in the pool, so slot identity stops drifting.

## Consecutive prefill batches

Two prefill iters with disjoint requests (e.g. `[A, B]` then `[C, D]`) write to disjoint slots; a later mixed/decode batch gathers `token_ids_buf[[A.idx, B.idx, C.idx, D.idx]]` and reads each request's first output independently — no overwrite, no combining logic.

## Chunked prefill

The same request keeps its `req_pool_idx` across all chunks. Mid-chunk writes to `token_ids_buf[req_pool_idx]` are never read (the request is in extend mode, input loaded from CPU); the final chunk's write is the real first output, picked up by gather the first time the request enters decode.
