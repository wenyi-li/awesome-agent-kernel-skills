## Summary

## Motivation

- [Efficiency] As shown in the benchmark scripts, switching to use PyArray deliver significant wins for long prompt scenarios (mainly coming from more efficient Tensor construction)
- [Almost-zero PyO3 overhead for future usage] Scheduler and Radix Cache is using PyList to communicate in nowadays; Once we migrate Radix Cache to Rust, PyList -> Rust::Vec PyO3 cost would become new bottleneck (while tree walk in Rust would be cheap); After the migration, benefits from the consecutive memory layout of PyArray, Python to Rust communcation cost would be extremely low (micro-second level).

## Modifications

Change fill_ids, original_input_ids and output_ids from PyList to PyArray for efficiency

## Accuracy Tests

We've verified llama 3.1 8B and Qwen 2.5 7B gsm8k correctness are on-par with trunk.

## Speed Tests and Profiling

E2E results: PyList vs PyArray is roughtly on-par in different lengths (minor regression in short prompt, and some improvement in long prompt)
