## Summary

## Motivation

Fix nightly `TestKimiK25.test_kimi_k25` failure (gsm8k 0.000 on both TP8 and TP8+DP8 since 2026-05-19, runs [26069709241](https://github.com/sgl-project/sglang/actions/runs/26069709241) and [26134864664](https://github.com/sgl-project/sglang/actions/runs/26134864664)).

Bisection points at #25390 (`abe2ec2af`), which widened the `n_routed_experts` allow-list in `determine_num_fused_shared_experts` from `{256}` to `{256, 384}` for AMD MXFP4. That unconditionally enables shared-experts fusion for **any** Kimi-K2.5 load. The fused path requires the shared expert to be pre-fused into the routed-experts tensor on disk — true only for `amd/Kimi-K2.5-MXFP4` (Quark). The standard `moonshotai/Kimi-K2.5` (compressed-tensors) checkpoint keeps the shared expert in `quantization_config.ignore` (different dtype/scale from the int4 routed experts) and the fused loader silently misses every `experts.wN_weight` (load log shows `not found in params_dict` for every MoE layer; weight mem 84.39 GB vs the safe 72.19 GB). Output is silent garbage.

## Modifications

Restrict the new `384` allow-list entry to `quant_config.get_name() == "quark"`. The `256` (DeepSeek-V3/R1) path is unchanged

## Problem

[Fix]: Restrict Kimi-K2.5 shared-experts fusion to Quark MXFP4 checkpoints
