## Summary

## Purpose

Batch vision encoder calls in `Gemma4ForConditionalGeneration` instead of
processing images/frames serially through the vision tower.

The baseline code calls the full vision tower (`vt(pv, pp, output_length)`)
once per image or video frame in a Python loop. Under concurrent serving load,
this serialization bottlenecks throughput — the encoder cannot overlap work
across items in the same batch.

This PR decomposes the monolithic `vt()` call into
`patch_embedder -> encoder -> pooler -> standardize -> embed_vision` and batches
each stage:

- Image path: groups images by patch count (resolution bucket) so each
  encoder call processes a uniform-shape batch with no cross-resolution
  padding. Pooling runs per-image (outputs are variable-length after padding
  removal). Projection (`embed_vision`, which is pointwise RMSNorm + Linear)
  runs over a single concatenated tensor.
- Video path: all frames across all videos are encoded together in
  memory-bounded chunks, then pooled per-frame and projected in one batched
  call. Regrouped into per-video tensors for `embed_multimodal`.
- Dynamic batch sizing: encoder batch ceiling is computed lazily on first
