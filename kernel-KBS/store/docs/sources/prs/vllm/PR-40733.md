## Summary

- Addressing the pain point 6 from here https://github.com/vllm-project/vllm/issues/32028
- vllm/distributed/eplb/rebalance_execute.py:586 had a device-wide GPU sync with a NOTE(@abmfy) comment admitting the original author didn't know why it was needed. After investigation, the line is dead code in the SYNC path (rearrange_expert_weights_inplace).

- The SYNC path runs entirely on the default CUDA stream end-to-end — torch.empty_like, move_to_buffer's b.copy_(w, non_blocking=True), and NCCL Send/Recv (default stream=None -> current_stream()) all share it. No cross-stream hazard exists. PyTorch's ProcessGroupNCCL correctly calls record_stream(), so the caching allocator is also safe.

- The ASYNC path (transfer_layer + async_worker) uses its own design — cuda_stream.synchronize() (async_worker.py:134) plus CpuGpuEvent for thread handoff (eplb_utils.py) — and is unaffected.

Verification
------------
- Stress (2x NVIDIA A100-PG509-200, SM80, torch_nccl, NCCL 2.28.9): 50 runs x 2000 iter x hidden=[1024,2048] = ~100k effective sync-path iterations -> 50/50 race-clean over 2.5h wall time.
