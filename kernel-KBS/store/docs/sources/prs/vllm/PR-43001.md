## Summary

## Purpose

Fixes #42995.

`/sleep?level>=1` and `/pause?clear_cache=true` go through `EngineCore` on the engine side, which only clears the **P1** multimodal receiver cache via `pause_scheduler -> _reset_caches -> reset_mm_cache`. The **P0** sender cache (`MultiModalProcessorSenderCache`) lives in the API server process and was never cleared on these paths. On the next request that reuses a previously-seen `mm_hash`, the sender hits, strips the payload, and forwards `mm_item=None` over IPC. P1 has no entry → `assert mm_item is not None` in `MultiModalReceiverCache.get_and_update_item` (`vllm/multimodal/cache.py:660`) trips, and after enough retries the EngineCore async loop wedges (only a process restart recovers).

> The gap was introduced by PR #34528, which centralized cache resetting in `pause_scheduler`. The `# we don't attempt to re-sync the internal caches (P0 sender, P1 receiver)` comment in `EngineCore.reset_mm_cache` was accurate when the method was only reachable via the debug `/reset_mm_cache` endpoint (which goes through `AsyncLLM.reset_mm_cache` and already dual-clears). It became wrong once `/sleep` and `/pause?clear_cache=true` reached the same code path impl

## Problem
