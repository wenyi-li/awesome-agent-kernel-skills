## Summary

Elastic EP changes the EP/DP topology after the model is already loaded. For
internal modular-kernel MoE methods, prepare_communication_buffer_for_model()
does not rebuild the kernel, so existing ranks keep a modular kernel that was
created for the old EP/DP configuration.

That breaks scale-up in two ways. First, existing ranks eventually need a
modular kernel for the new EP/DP configuration. Second, new ranks create their
internal MK in load_model(), and MK creation creates a prepare/finalize object
that may perform collective EP communicator setup. Existing ranks must
participate in that collective while the new ranks are loading; waiting until
SWITCH_AND_PREPARE leaves the new ranks hanging.

Add an EEP quant method stage/commit path. Existing ranks stage a new quant
method object against the standby EP group while new ranks load_model(). Staging
creates a new prepare/finalize object, recreates the compatible experts object,
builds a new FusedMoEKernel, and wraps that kernel in a FusedMoEModularMethod
quant method wrapper. The staged quant method is then committed when we switch
to the new configuration.
