## Summary

PR #39805 taught the V1 model runner to call `get_language_model()` before `is_mixture_of_experts()` so EPLB works on VLMs like `KimiK25ForConditionalGeneration`, where the top-level wrapper doesn't implement the `MixtureOfExperts` protocol itself.

The V2 path in `vllm/v1/worker/gpu/eplb_utils.py` (added in #37488, before that V1 fix) never picked up the unwrap. So on V2 with `--enable-eplb`:

- `maybe_register_model()` gets handed the raw wrapper, the protocol check fails, `_has_registered_models` stays `False`, and `step()` no-ops for the rest of the run — the flag is silently dropped.
- `setup_from_mapping()` has the same check as an `assert`, so an elastic-EP remap on a VLM-MoE model would crash with `AssertionError` instead of just being skipped.

This just mirrors the V1 unwrap. Pulled it into a tiny helper so both sites can share it.

`Llama4ForConditionalGeneration` isn't affected because it copies the MoE protocol attributes onto the wrapper (see `mllama4.py:788-796`). The bug only hits wrappers that don't do that proxying — `KimiK25` is the one I confirmed.
