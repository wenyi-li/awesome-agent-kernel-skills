## Summary

## Purpose

- Add a new CLI argument `--cpu-distributed-timeout-seconds` to independently configure the timeout for CPU (gloo) process groups, decoupled from the existing device timeout option `--distributed-timeout-seconds`.
- Improve fault tolerance behavior in future elastic / failure-recovery deployments, where the default Gloo timeout (1800s) can lead to long blocking periods (up to 30 minutes) when a rank fails.
- Maintain full backward compatibility: when the flag is not set, PyTorch’s default Gloo timeout (1800s) remains unchanged.

## Motivation

vLLM initializes both device (NCCL) and CPU (gloo) process groups for distributed execution. The existing `--distributed-timeout-seconds` option only applies to device-based communication (see: https://github.com/vllm-project/vllm/pull/36047). However, CPU process groups—used for CPU-side coordination such as metadata exchange—still rely on PyTorch’s hardcoded default timeout of 1800 seconds.

By introducing `--cpu-distributed-timeout-seconds`, operators can configure a much shorter timeout (e.g., 5–10 seconds), allowing faster recovery in the future without affecting NCCL timeout semantics.
