# AutoDD And Debug Tools Advanced

This page covers AutoDD internals, runner behavior, generated-source callbacks,
plot options, and the experimental static analyzer.

## AutoDD Reduction Model

AutoDD is AST-driven in the active command path. It tries reducer groups such
as:

- removing statements and expressions
- simplifying `if` branches
- binding loop variables to zero-like values
- forwarding call arguments
- simplifying binary operators
- removing function arguments or keywords
- reducing integer constants larger than one by halving

A candidate is accepted when the configured error substring appears in stdout
or stderr. Accepted candidates are cleaned where possible and written to the
output file.

## Runner Backends

`--backend runner` uses a persistent Python worker. It is faster because the
worker process is reused.

`--backend subproc` launches a subprocess per candidate. It is slower but gives
better isolation when the failing program can poison process state.

Use `subproc` when the reproducer crashes native code, leaves CUDA state
unusable, or mutates global process state in a way that affects later
candidates.

## Freeze Boundaries

AutoDD recognizes:

- `with __freeze__:` blocks
- `__freeze__(expr)` calls
- `# autodd: freeze` single-line annotations
- `# autodd: freeze-start` / `# autodd: end-freeze` blocks

Frozen regions are removed from the rewrite-label set. This protects imports,
device setup, data construction, or reference checks needed to preserve the
failure.

## Generated-Source Postprocessing

Top-level callback registration hooks let tooling modify generated source after
TileLang lowering:

```python
def add_marker(src: str) -> str:
    return "// generated by debug run\n" + src

tilelang.register_cuda_postproc(add_marker)
tilelang.register_hip_postproc(add_marker)
tilelang.register_c_postproc(add_marker)
```

Use these hooks sparingly. They affect generated source globally in the current
process, so they are best for instrumentation, build-system integration, or
temporary debugging.

## Plot Layout Options

`plot_layout(...)` accepts names, output formats, save directory, and view
controls. For `T.Layout`, use `view="input"` to label output coordinates from
input cells, or `view="output"` to invert the map.

For fragments, the plot path shows thread/value ownership. This is useful when
checking MMA/MFMA fragment layouts, swizzles, and transforms. Very large layouts
produce crowded figures; reduce the shape before plotting when possible.

## `plot_fragment_tv`

The plotting module also exposes a lower-level `plot_fragment_tv(...)` helper
for thread/value fragment views. Prefer `plot_layout(...)` unless you are
working directly with the thread/value representation.

## Static Analyzer

`tilelang.tools.Analyzer` is exported through `tilelang.tools`. It walks a
TileLang function and estimates memory traffic and arithmetic work for selected
operations such as copy and GEMM. The public convenience form is:

```python
from tilelang.tools import Analyzer

result = Analyzer.analysis(func, device="cuda")
print(result)
```

Treat the analyzer as a static estimate, not a profiler. It does not replace
generated-source inspection or runtime benchmarking.

## Debug Checklist For Large Failures

1. Reproduce from a standalone Python script.
2. Capture the shortest stable error substring.
3. Freeze imports, setup, and input construction that must remain.
4. Run AutoDD with one worker first.
5. Switch to `--backend subproc` if process state becomes unreliable.
6. Inspect the minimized file before filing or preserving a repro.

For layout bugs, first create a tiny `T.Layout` or `T.Fragment` example and
plot it directly. Use compile-time layout visualization only when the issue
depends on inferred layouts inside a real kernel.
